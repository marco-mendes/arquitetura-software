## O Estilo Arquitetural em Camadas

> **Localização no mapa arquitetural:** Camadas pertence à **Família 1 — Monolítica** apresentada em *[1.1 — Mapa e Estilos de Backend](1.1%20Mapa%20e%20Estilos%20de%20Backend.md)*. Esta seção aprofunda o funcionamento interno, variações e exemplos de código do estilo.

### O que é o estilo arquitetural em camadas?

O estilo arquitetural em camadas é um dos mais difundidos e intuitivos da engenharia de software. Sua organização baseia-se na separação de responsabilidades por "tipos de tarefas" realizadas na aplicação, agrupando funcionalidades em camadas que se sobrepõem logicamente.

Trata-se de um estilo técnico, geralmente não orientado a domínio. Em muitos contextos, esse estilo surge mesmo sem planejamento, sendo o resultado natural da estrutura da equipe e da forma como os sistemas vão sendo construídos incrementalmente. Esse fenômeno é descrito no livro *Fundamentals of Software Architecture* como um dos sintomas da *"architecture by implication"* e está relacionado com a **Lei de Conway**: organizações que projetam sistemas estão limitadas a produzir projetos que são cópias da estrutura de comunicação dessas organizações.

<img width="481" alt="image" src="https://github.com/user-attachments/assets/682e0c79-dd5d-428a-9ce8-60e39d9fd4f4" />

A arquitetura em camadas organiza logicamente os componentes da aplicação em blocos horizontais. Cada camada é responsável por uma parte do processamento. A divisão mais comum no backend é:

* **Camada de Apresentação:** expõe o sistema ao mundo externo — endpoints HTTP, serialização de respostas, validação de entrada.
* **Camada de Negócios:** contém as regras que definem o comportamento da aplicação, independente de como os dados chegam ou onde são armazenados.
* **Camada de Dados:** gerencia a recuperação e persistência de dados, isolando os detalhes do banco do restante do sistema.

## O Princípio Aberto-Fechado no Estilo em Camadas

O **princípio aberto-fechado** afirma que os componentes do sistema devem ser projetados para serem **abertos para extensões**, mas **fechados para modificações**. No contexto do estilo em camadas, esse princípio garante que cada camada possa ser estendida sem que seja necessário modificar o comportamento das outras camadas.

Considere a camada de dados de um sistema que inicialmente usa **MySQL**. Para migrar para **PostgreSQL** sem impactar a camada de negócios, basta criar uma abstração que ambas as implementações satisfazem:

```typescript
// Abstração da camada de dados
interface RepositorioProduto {
  salvar(nome: string, preco: number): void;
  listar(): { nome: string; preco: number }[];
}

// Implementação para MySQL
class MySQLRepositorioProduto implements RepositorioProduto {
  private produtos: { nome: string; preco: number }[] = [];

  salvar(nome: string, preco: number): void {
    this.produtos.push({ nome, preco });
    console.log("Produto salvo no MySQL.");
  }

  listar(): { nome: string; preco: number }[] {
    console.log("Listando produtos do MySQL.");
    return this.produtos;
  }
}

// Implementação para PostgreSQL
class PostgreSQLRepositorioProduto implements RepositorioProduto {
  private produtos: { nome: string; preco: number }[] = [];

  salvar(nome: string, preco: number): void {
    this.produtos.push({ nome, preco });
    console.log("Produto salvo no PostgreSQL.");
  }

  listar(): { nome: string; preco: number }[] {
    console.log("Listando produtos do PostgreSQL.");
    return this.produtos;
  }
}

// Camada de negócios: não conhece o banco, apenas a abstração
class ProdutoServico {
  constructor(private repositorio: RepositorioProduto) {}

  adicionarProduto(nome: string, preco: number): void {
    this.repositorio.salvar(nome, preco);
  }

  exibirProdutos(): void {
    const produtos = this.repositorio.listar();
    produtos.forEach(p => console.log(`Produto: ${p.nome}, Preço: R$${p.preco}`));
  }
}
```

A classe `ProdutoServico` (camada de negócios) não muda quando o banco muda. Isso é o OCP aplicado à fronteira entre camadas.

## Variações do Estilo em Camadas no Backend

### MVC (Model-View-Controller)

O padrão MVC é a especialização mais popular do estilo em camadas para o ciclo requisição-resposta HTTP. Foi proposto por Trygve Reenskaug em 1979 no Xerox PARC e consolidado em frameworks web como **Ruby on Rails**, **ASP.NET MVC**, **Spring MVC** e **Laravel**.

<img width="300" alt="image" src="https://github.com/user-attachments/assets/c2e3654b-307e-447c-9834-5a4cffabdfe5" />

No contexto de **backend**, os três componentes têm papéis bem definidos:

* **Model:** representa os dados e as regras de negócio. Encapsula o estado da aplicação e é responsável por manter invariantes do domínio.
* **View:** formata e serializa a resposta — JSON, XML ou HTML gerado no servidor. Em APIs REST modernas, o "V" frequentemente é a camada de serialização.
* **Controller:** recebe a requisição HTTP, valida a entrada, coordena a execução da lógica no Model e devolve a resposta via View.

**Aplicações típicas de backend com MVC:**
- Sistemas bancários e financeiros
- Aplicações de comércio eletrônico
- Sistemas de gestão hospitalar
- Portais de serviços governamentais

**Frameworks de backend que implementam MVC:**

| Camada | Tecnologias |
|--------|-------------|
| Controller | ASP.NET MVC, Spring MVC, Ruby on Rails, Laravel, Django |
| Model / Negócios | .NET Core, Spring Boot, Node.js (Express) |
| Dados | Entity Framework, Hibernate, ActiveRecord, Eloquent |

#### Exemplo de código — MVC no Backend

```typescript
// Model: encapsula dados e regras de negócio
class Produto {
  constructor(public id: number, public nome: string, public preco: number) {}

  aplicarDesconto(percentual: number): void {
    this.preco -= this.preco * (percentual / 100);
  }
}

// View: serializa a resposta (equivale ao JSON de retorno numa API)
class ProdutoView {
  serializar(produto: Produto): string {
    return JSON.stringify({ id: produto.id, nome: produto.nome, preco: produto.preco.toFixed(2) });
  }
}

// Controller: orquestra a requisição
class ProdutoController {
  constructor(private produto: Produto, private view: ProdutoView) {}

  aplicarDesconto(percentual: number): string {
    this.produto.aplicarDesconto(percentual);
    return this.view.serializar(this.produto);
  }
}

// Uso
const produto = new Produto(1, "Notebook", 5000);
const view = new ProdutoView();
const controller = new ProdutoController(produto, view);

console.log(controller.aplicarDesconto(10));
// {"id":1,"nome":"Notebook","preco":"4500.00"}
```

### DDD (Domain-Driven Design)

O DDD é uma abordagem para organizar o backend de sistemas com domínio de negócio complexo. Diferente do MVC — que é organizado por *tipo técnico* (Model, View, Controller) — o DDD organiza o código por *conceitos de negócio* (Pedido, Pagamento, Cliente).

<img width="300" alt="image" src="https://github.com/user-attachments/assets/90a53c69-9993-4cd0-bc82-5b65dd17fbb0" />

Os blocos construtivos centrais do DDD na camada de negócios:

* **Entidades:** objetos com identidade persistente e regras de negócio próprias.
* **Value Objects:** descrevem aspectos do domínio sem identidade própria (ex: `Dinheiro`, `Endereco`).
* **Agregados:** grupos de entidades com consistência transacional gerenciada por uma Raiz.
* **Repositórios:** abstraem a persistência dos agregados — a camada de negócios não conhece o banco.

#### Exemplo de código — Agregado de Pedido

```typescript
// Value Object: imutável, sem identidade própria
class Dinheiro {
  constructor(public readonly valor: number, public readonly moeda: string = "BRL") {
    if (valor < 0) throw new Error("Valor monetário não pode ser negativo.");
  }

  somar(outro: Dinheiro): Dinheiro {
    if (this.moeda !== outro.moeda) throw new Error("Moedas diferentes.");
    return new Dinheiro(this.valor + outro.valor, this.moeda);
  }
}

// Entidade com regras de negócio
class Produto {
  constructor(public readonly id: number, public nome: string, public preco: Dinheiro) {}

  alterarPreco(novoPreco: Dinheiro): void {
    if (novoPreco.valor < 1) throw new Error("Preço mínimo é R$1,00.");
    this.preco = novoPreco;
  }
}

// Entidade interna do agregado
class ItemPedido {
  constructor(public readonly produto: Produto, public readonly quantidade: number) {
    if (quantidade <= 0) throw new Error("Quantidade deve ser positiva.");
  }

  get total(): Dinheiro {
    return new Dinheiro(this.produto.preco.valor * this.quantidade, this.produto.preco.moeda);
  }
}

// Agregado Raiz: controla consistência de todos os itens
class Pedido {
  private itens: ItemPedido[] = [];
  private _valorTotal: Dinheiro = new Dinheiro(0);
  private readonly MAX_ITENS = 10;

  constructor(public readonly id: number, public readonly clienteId: number) {}

  adicionarItem(produto: Produto, quantidade: number): void {
    if (this.itens.length >= this.MAX_ITENS) {
      throw new Error("Pedido não pode ter mais de 10 itens.");
    }
    const item = new ItemPedido(produto, quantidade);
    this.itens.push(item);
    this._valorTotal = this._valorTotal.somar(item.total);
  }

  get valorTotal(): Dinheiro { return this._valorTotal; }
  listarItens(): ItemPedido[] { return [...this.itens]; }
}

// Repositório: abstrai a persistência (a camada de negócios não sabe onde o dado fica)
class PedidoRepositorio {
  private pedidos: Pedido[] = [];

  salvar(pedido: Pedido): void {
    this.pedidos.push(pedido);
  }

  buscarPorId(id: number): Pedido | undefined {
    return this.pedidos.find(p => p.id === id);
  }
}

// Uso
const produto1 = new Produto(1, "Notebook", new Dinheiro(5000));
const produto2 = new Produto(2, "Mouse", new Dinheiro(150));
const pedido = new Pedido(1, 101);

pedido.adicionarItem(produto1, 1);
pedido.adicionarItem(produto2, 2);

const repo = new PedidoRepositorio();
repo.salvar(pedido);

console.log("Total:", pedido.valorTotal.valor, pedido.valorTotal.moeda); // 5300 BRL
```

**Quando preferir DDD a Camadas simples:** quando as regras de negócio são complexas, mudam com frequência e o maior risco do projeto é o mal-entendido entre time técnico e especialistas de negócio. Para sistemas CRUD simples, o overhead do DDD não se justifica.

## Camadas Lógicas e Camadas Físicas

A arquitetura em camadas define uma estrutura **lógica**, mas a implantação **física** pode variar amplamente conforme o porte da aplicação e os requisitos não funcionais como escalabilidade, desempenho e segurança.

### 1. Uma camada física (monolito)

<img width="400" alt="image" src="https://github.com/user-attachments/assets/60f789a1-aca0-4759-8ba9-259bfff52efc" />

Todas as camadas lógicas empacotadas em um único executável, implantado em um único servidor.

**Exemplo:** aplicação Spring Boot com banco H2 embutido para um sistema interno de RH.

| | |
|---|---|
| **Vantagens** | Simplicidade, menor custo, fácil de depurar e empacotar |
| **Desvantagens** | Difícil de escalar componentes individualmente, alto risco em alterações |

---

### 2. Duas camadas físicas (cliente + servidor)

<img width="400" alt="image" src="https://github.com/user-attachments/assets/1a7430c2-266d-4005-9b71-ebec7dff1f42" />

A camada de apresentação do backend (API REST) fica separada da camada de dados. O cliente (mobile, browser) consome a API.

**Exemplo:** API Node.js com regras de negócio e banco PostgreSQL em servidor separado.

| | |
|---|---|
| **Vantagens** | Backend e banco escalam e evoluem de forma independente |
| **Desvantagens** | Latência de rede entre camadas, necessidade de gerenciar dois serviços |

---

### 3. Três ou mais camadas físicas

<img width="400" alt="image" src="https://github.com/user-attachments/assets/5170effe-c89f-47a3-9bc9-41095836dba0" />

Cada camada lógica roda em processo ou servidor isolado: a camada de negócios exposta via API interna (REST ou gRPC), a camada de dados em serviço separado com controles de acesso rígidos.

**Exemplo:** API C# (apresentação) → serviço .NET Core com regras (negócios) → banco Oracle em rede isolada (dados).

| | |
|---|---|
| **Vantagens** | Escalabilidade fina por camada, isolamento de falhas, times independentes por camada |
| **Desvantagens** | Complexidade operacional, latência acumulada, necessidade de observabilidade |

---

> **Atenção:** distribuir camadas fisicamente não resolve problemas de acoplamento mal definidos. Um sistema com responsabilidades misturadas entre camadas continua acoplado mesmo rodando em servidores separados. A estrutura lógica precisa estar sólida antes de qualquer decisão de distribuição física.
