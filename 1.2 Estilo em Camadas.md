# Estilo em Camadas (Layered Architecture)

> **Localização no mapa arquitetural:** Camadas pertence à **Família 1 — Monolítica** apresentada em *[1.1 — Mapa e Estilos de Backend](1.1%20Mapa%20e%20Estilos%20de%20Backend.md)*. Compare com [1.3 — Pipes and Filters](1.3%20Pipes-filters.md) quando o problema central for processamento sequencial de dados, e com [1.4 — MicroKernel](1.4%20Micro-kernel.md) quando extensibilidade do produto for o requisito dominante.

## O que é o estilo arquitetural em camadas?

O estilo arquitetural em camadas é um dos mais difundidos e intuitivos da engenharia de software. Sua organização baseia-se na separação de responsabilidades por **tipo técnico de tarefa**, agrupando funcionalidades em camadas horizontais com comunicação controlada entre elas.

Trata-se de um estilo **técnico**, geralmente não orientado a domínio. Em muitos contextos, esse estilo surge mesmo sem planejamento — é o resultado natural da estrutura da equipe e da forma como os sistemas crescem organicamente. Mark Richards e Neal Ford descrevem esse fenômeno no livro *Fundamentals of Software Architecture* como *"architecture by implication"*: o estilo emerge por inércia, não por decisão. A **Lei de Conway** reforça essa tendência — organizações que projetam sistemas produzem arquiteturas que espelham sua própria estrutura de comunicação.

<img width="481" alt="image" src="https://github.com/user-attachments/assets/682e0c79-dd5d-428a-9ce8-60e39d9fd4f4" />

A divisão mais comum no backend é:

* **Camada de Apresentação:** expõe o sistema ao mundo externo — endpoints HTTP, serialização de respostas, validação de entrada.
* **Camada de Negócios:** contém as regras que definem o comportamento da aplicação, independente de como os dados chegam ou onde são armazenados.
* **Camada de Dados:** gerencia a recuperação e persistência de dados, isolando os detalhes do banco do restante do sistema.

## Características Arquiteturais

Richards e Ford avaliam estilos arquiteturais a partir de um conjunto padronizado de características. Para o estilo em Camadas (Fundamentals of Software Architecture, Cap. 10):

| Característica | Avaliação | Observação |
|---|---|---|
| Custo geral | ⭐⭐⭐⭐⭐ | Baixo custo de entrada; tecnologia amplamente conhecida |
| Simplicidade | ⭐⭐⭐⭐⭐ | Fácil de compreender e implementar |
| Escalabilidade | ⭐⭐☆☆☆ | Escala como unidade única; difícil escalar componentes individualmente |
| Elasticidade | ⭐☆☆☆☆ | Pouca capacidade de expansão/retração rápida sob carga variável |
| Implantabilidade | ⭐⭐☆☆☆ | Uma unidade de implantação; qualquer mudança implanta tudo |
| Testabilidade | ⭐⭐⭐☆☆ | Camadas podem ser testadas em isolamento via mocks |
| Desempenho | ⭐⭐⭐☆☆ | Adequado para a maioria dos casos; sobrecarga em requisições que atravessam muitas camadas |
| Modularidade | ⭐⭐☆☆☆ | Modular logicamente, mas acoplado fisicamente |
| Confiabilidade | ⭐⭐⭐☆☆ | Falha de um componente pode afetar todo o sistema |

> **Leitura recomendada:** Richards, M.; Ford, N. *Fundamentals of Software Architecture*, 2ª ed., Cap. 10 — Layered Architecture Style.

## Camadas Abertas e Fechadas

Richards e Ford introduzem uma distinção crítica entre **camadas fechadas** e **camadas abertas** que define o comportamento do estilo.

Uma **camada fechada** exige que toda requisição passe por ela antes de chegar à próxima camada — mesmo que não haja lógica útil a executar. Uma **camada aberta** permite que uma requisição "pule" diretamente para a camada seguinte.

```
Fechada:   [Apresentação] → [Negócios] → [Persistência] → [Banco]
Aberta:    [Apresentação] → → → → → → → [Banco]  (acesso direto permitido)
```

**Camadas fechadas** garantem **isolamento** — mudanças numa camada não vazam para as outras. O custo é rigidez: toda requisição, mesmo simples, percorre todas as camadas.

**Camadas abertas** oferecem **flexibilidade** — componentes podem acessar qualquer camada abaixo. O custo é acoplamento difuso: o isolamento original se perde com o tempo, criando dependências que dificultam mudanças futuras.

A decisão de abrir ou fechar cada camada deve ser documentada e justificada — é exatamente o tipo de decisão que um ADR deve registrar (ver [1.5 — Decisões Arquiteturais](1.5%20O%20Racional%20arquitetural%20e%20o%20conceito%20de%20ADRs.md)).

## O Anti-padrão do Sumidouro

Richards e Ford identificam o **Architecture Sinkhole** como o anti-padrão mais comum do estilo em camadas. Ele ocorre quando uma requisição simplesmente passa por todas as camadas **sem que nenhuma lógica relevante seja executada** em cada uma delas.

```
[GET /usuarios] → Apresentação (só repassa)
               → Negócios    (só repassa)
               → Dados       (SELECT * FROM usuarios)
               ← ← ← ← ← ← retorna resultado sem transformação
```

O problema não é que essa requisição exista — consultas simples são legítimas. O problema é quando **a maioria** das requisições do sistema segue esse padrão. Nesse caso, as camadas intermediárias acrescentam latência e complexidade sem valor arquitetural.

> **Regra prática de Richards e Ford:** se mais de 80% das requisições do sistema são sumidouros, o estilo em camadas provavelmente não é o certo para esse sistema. Considere um estilo com menos indireções, ou consolide as camadas desnecessárias.

## O Princípio Aberto-Fechado na Camada de Dados

O **princípio aberto-fechado (OCP)** afirma que os componentes devem ser **abertos para extensão**, mas **fechados para modificação**. Na fronteira entre a camada de negócios e a camada de dados, isso se traduz em abstrair o acesso ao banco por meio de interfaces.

Considere a migração de **MySQL** para **PostgreSQL** sem impactar a camada de negócios:

```typescript
// Abstração da camada de dados
interface RepositorioProduto {
  salvar(nome: string, preco: number): void;
  listar(): { nome: string; preco: number }[];
}

class MySQLRepositorioProduto implements RepositorioProduto {
  private produtos: { nome: string; preco: number }[] = [];

  salvar(nome: string, preco: number): void {
    this.produtos.push({ nome, preco });
    console.log("Produto salvo no MySQL.");
  }

  listar(): { nome: string; preco: number }[] {
    console.log("Listando produtos do MySQL.");
    return this.produtos;
  }
}

class PostgreSQLRepositorioProduto implements RepositorioProduto {
  private produtos: { nome: string; preco: number }[] = [];

  salvar(nome: string, preco: number): void {
    this.produtos.push({ nome, preco });
    console.log("Produto salvo no PostgreSQL.");
  }

  listar(): { nome: string; preco: number }[] {
    console.log("Listando produtos do PostgreSQL.");
    return this.produtos;
  }
}

// Camada de negócios: depende da abstração, não da implementação
class ProdutoServico {
  constructor(private repositorio: RepositorioProduto) {}

  adicionarProduto(nome: string, preco: number): void {
    this.repositorio.salvar(nome, preco);
  }

  exibirProdutos(): void {
    const produtos = this.repositorio.listar();
    produtos.forEach(p => console.log(`Produto: ${p.nome}, Preço: R$${p.preco}`));
  }
}
```

`ProdutoServico` não muda quando o banco muda — apenas a implementação de `RepositorioProduto` é substituída. Isso é o OCP aplicado à fronteira entre camadas.

## Variações do Estilo em Camadas no Backend

### MVC (Model-View-Controller)

O padrão MVC é a especialização mais popular do estilo em camadas para o ciclo requisição-resposta HTTP. Foi proposto por Trygve Reenskaug em 1979 e consolidado em frameworks web como **Ruby on Rails**, **ASP.NET MVC**, **Spring MVC** e **Laravel**.

<img width="300" alt="image" src="https://github.com/user-attachments/assets/c2e3654b-307e-447c-9834-5a4cffabdfe5" />

No contexto de **backend**, os três componentes têm papéis bem definidos:

* **Model:** representa os dados e as regras de negócio. Encapsula o estado da aplicação e mantém invariantes do domínio.
* **View:** formata e serializa a resposta — JSON, XML ou HTML gerado no servidor.
* **Controller:** recebe a requisição HTTP, valida a entrada, coordena a execução da lógica no Model e devolve a resposta via View.

**Frameworks de backend que implementam MVC:**

| Camada | Tecnologias |
|--------|-------------|
| Controller | ASP.NET MVC, Spring MVC, Ruby on Rails, Laravel, Django |
| Model / Negócios | .NET Core, Spring Boot, Node.js (Express) |
| Dados | Entity Framework, Hibernate, ActiveRecord, Eloquent |

#### Exemplo — MVC no Backend

```typescript
// Model: encapsula dados e regras de negócio
class Produto {
  constructor(public id: number, public nome: string, public preco: number) {}

  aplicarDesconto(percentual: number): void {
    this.preco -= this.preco * (percentual / 100);
  }
}

// View: serializa a resposta (equivale ao JSON de retorno numa API REST)
class ProdutoView {
  serializar(produto: Produto): string {
    return JSON.stringify({ id: produto.id, nome: produto.nome, preco: produto.preco.toFixed(2) });
  }
}

// Controller: orquestra a requisição
class ProdutoController {
  constructor(private produto: Produto, private view: ProdutoView) {}

  aplicarDesconto(percentual: number): string {
    this.produto.aplicarDesconto(percentual);
    return this.view.serializar(this.produto);
  }
}

const produto = new Produto(1, "Notebook", 5000);
const controller = new ProdutoController(produto, new ProdutoView());
console.log(controller.aplicarDesconto(10));
// {"id":1,"nome":"Notebook","preco":"4500.00"}
```

### DDD (Domain-Driven Design)

O DDD organiza o backend por **conceitos de negócio** em vez de tipo técnico. Diferente do MVC — que é estruturado em Model, View, Controller — o DDD estrutura o código em Pedido, Pagamento, Cliente e seus relacionamentos.

<img width="300" alt="image" src="https://github.com/user-attachments/assets/90a53c69-9993-4cd0-bc82-5b65dd17fbb0" />

Os blocos construtivos do DDD na camada de negócios:

* **Entidades:** objetos com identidade persistente e regras de negócio próprias.
* **Value Objects:** descrevem aspectos do domínio sem identidade própria (ex: `Dinheiro`, `Endereco`).
* **Agregados:** grupos de entidades com consistência transacional gerenciada por uma Raiz.
* **Repositórios:** abstraem a persistência dos agregados — a camada de negócios não conhece o banco.

#### Exemplo — Agregado de Pedido

```typescript
class Dinheiro {
  constructor(public readonly valor: number, public readonly moeda: string = "BRL") {
    if (valor < 0) throw new Error("Valor monetário não pode ser negativo.");
  }

  somar(outro: Dinheiro): Dinheiro {
    if (this.moeda !== outro.moeda) throw new Error("Moedas diferentes.");
    return new Dinheiro(this.valor + outro.valor, this.moeda);
  }
}

class Produto {
  constructor(public readonly id: number, public nome: string, public preco: Dinheiro) {}

  alterarPreco(novoPreco: Dinheiro): void {
    if (novoPreco.valor < 1) throw new Error("Preço mínimo é R$1,00.");
    this.preco = novoPreco;
  }
}

class ItemPedido {
  constructor(public readonly produto: Produto, public readonly quantidade: number) {
    if (quantidade <= 0) throw new Error("Quantidade deve ser positiva.");
  }

  get total(): Dinheiro {
    return new Dinheiro(this.produto.preco.valor * this.quantidade, this.produto.preco.moeda);
  }
}

class Pedido {
  private itens: ItemPedido[] = [];
  private _valorTotal: Dinheiro = new Dinheiro(0);
  private readonly MAX_ITENS = 10;

  constructor(public readonly id: number, public readonly clienteId: number) {}

  adicionarItem(produto: Produto, quantidade: number): void {
    if (this.itens.length >= this.MAX_ITENS) {
      throw new Error("Pedido não pode ter mais de 10 itens.");
    }
    const item = new ItemPedido(produto, quantidade);
    this.itens.push(item);
    this._valorTotal = this._valorTotal.somar(item.total);
  }

  get valorTotal(): Dinheiro { return this._valorTotal; }
  listarItens(): ItemPedido[] { return [...this.itens]; }
}

class PedidoRepositorio {
  private pedidos: Pedido[] = [];

  salvar(pedido: Pedido): void { this.pedidos.push(pedido); }
  buscarPorId(id: number): Pedido | undefined {
    return this.pedidos.find(p => p.id === id);
  }
}
```

**Quando preferir DDD a Camadas simples:** quando as regras de negócio são complexas, mudam com frequência e o risco dominante é o mal-entendido entre o time técnico e os especialistas de negócio. Para sistemas CRUD simples, o overhead do DDD não se justifica.

## Camadas Lógicas e Camadas Físicas

A arquitetura em camadas define uma estrutura **lógica**, mas a implantação **física** pode variar amplamente.

### 1. Uma camada física (monolito)

<img width="400" alt="image" src="https://github.com/user-attachments/assets/60f789a1-aca0-4759-8ba9-259bfff52efc" />

Todas as camadas lógicas empacotadas num único executável e servidor.

**Exemplo:** aplicação Spring Boot com banco H2 embutido para um sistema interno.

| | |
|---|---|
| **Vantagens** | Simplicidade, menor custo, fácil de depurar |
| **Desvantagens** | Difícil escalar componentes individualmente; risco alto em deploys |

### 2. Duas camadas físicas (cliente + servidor)

<img width="400" alt="image" src="https://github.com/user-attachments/assets/1a7430c2-266d-4005-9b71-ebec7dff1f42" />

API REST separada do banco de dados.

**Exemplo:** API Node.js com regras de negócio conectando a PostgreSQL em servidor separado.

| | |
|---|---|
| **Vantagens** | Backend e banco evoluem e escalam de forma independente |
| **Desvantagens** | Latência de rede entre camadas; gestão de dois serviços |

### 3. Três ou mais camadas físicas

<img width="400" alt="image" src="https://github.com/user-attachments/assets/5170effe-c89f-47a3-9bc9-41095836dba0" />

Cada camada lógica roda em processo ou servidor isolado.

**Exemplo:** API C# (apresentação) → serviço .NET Core com regras (negócios) → banco Oracle em rede isolada (dados).

| | |
|---|---|
| **Vantagens** | Escalabilidade fina por camada; isolamento de falhas; times independentes |
| **Desvantagens** | Complexidade operacional; latência acumulada; exige observabilidade |

> **Atenção:** distribuir camadas fisicamente não resolve problemas de acoplamento lógico mal definido. Um sistema com responsabilidades misturadas entre camadas continua acoplado mesmo rodando em servidores separados.

## Quando usar — e quando não usar

**Use Camadas quando:**
- O sistema tem escopo e regras de negócio bem definidos
- A equipe é pequena e o orçamento é limitado
- Velocidade de desenvolvimento inicial é mais importante que escalabilidade
- O sistema é predominantemente CRUD com lógica de negócio moderada

**Não use Camadas quando:**
- Alta escalabilidade ou elasticidade são requisitos não negociáveis
- A maioria das requisições são sumidouros — o estilo está acrescentando latência sem valor
- Times independentes precisam implantar partes do sistema de forma autônoma (considere Microsserviços)
- O processamento é predominantemente orientado a fluxo de dados (considere [Pipes and Filters — 1.3](1.3%20Pipes-filters.md))
- Extensibilidade do produto para diferentes mercados é o driver central (considere [MicroKernel — 1.4](1.4%20Micro-kernel.md))
