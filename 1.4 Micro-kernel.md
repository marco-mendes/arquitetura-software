# Estilo MicroKernel (Microkernel Architecture)

> **Localização no mapa arquitetural:** MicroKernel pertence à **Família 1 — Monolítica** apresentada em *[1.1 — Mapa e Estilos de Backend](1.1%20Mapa%20e%20Estilos%20de%20Backend.md)*. Compare com [1.2 — Camadas](1.2%20Estilo%20em%20Camadas.md) quando a organização por responsabilidade técnica for suficiente, e com [1.3 — Pipes and Filters](1.3%20Pipes-filters.md) quando o problema central for transformação sequencial de dados.

## O que é o estilo MicroKernel?

O estilo **MicroKernel** — também chamado de *Plug-in Architecture* por Mark Richards e Neal Ford — separa o sistema em dois tipos de componentes: um **núcleo** estável com funcionalidade mínima e **plugins** independentes que estendem essa funcionalidade sem modificar o núcleo.

<img width="1110" alt="image" src="https://github.com/user-attachments/assets/21fb2fbd-92e2-4ca9-a5f5-b25b0f633af1" />

A premissa fundamental é que o **núcleo evolui lentamente** enquanto os **plugins evoluem rapidamente** — e essa evolução acontece de forma independente, sem coordenação entre as equipes que desenvolvem cada plugin. O núcleo não sabe o que os plugins fazem; os plugins não sabem uns dos outros. A única relação é o **contrato** que o núcleo define e que cada plugin implementa.

Esse estilo resolve o problema de produtos de software que precisam se adaptar a diferentes clientes, mercados ou regulamentações sem que o código central seja modificado a cada customização. É o padrão por trás de IDEs como Eclipse e VS Code, servidores de aplicação como WildFly e Tomcat, ferramentas de CI/CD como Jenkins, e sistemas de ERP configuráveis.

## Características Arquiteturais

Richards e Ford avaliam o estilo MicroKernel em *Fundamentals of Software Architecture* (Cap. 12):

| Característica | Avaliação | Observação |
|---|---|---|
| Custo geral | ⭐⭐⭐☆☆ | Custo médio; exige design cuidadoso do contrato do núcleo |
| Simplicidade | ⭐⭐⭐☆☆ | Conceito simples; implementação do contrato requer disciplina |
| Escalabilidade | ⭐⭐⭐☆☆ | Escala como unidade; plugins podem ser carregados/descarregados |
| Elasticidade | ⭐⭐⭐☆☆ | Similar à escalabilidade |
| Implantabilidade | ⭐⭐⭐☆☆ | Plugins podem ser implantados independentemente do núcleo |
| Testabilidade | ⭐⭐⭐☆☆ | Núcleo é fácil de testar; plugins requerem testes de integração com o núcleo |
| Desempenho | ⭐⭐⭐☆☆ | Overhead de dispatch para plugins; geralmente aceitável |
| Modularidade | ⭐⭐⭐⭐☆ | **Ponto forte:** plugins completamente independentes entre si |
| Confiabilidade | ⭐⭐⭐☆☆ | Plugin com falha pode ser isolado sem derrubar o núcleo |

> **Leitura recomendada:** Richards, M.; Ford, N. *Fundamentals of Software Architecture*, 2ª ed. O'Reilly, 2022. Cap. 12 — Microkernel Architecture Style.

## Os Três Componentes Principais

Richards e Ford definem três componentes estruturais do estilo:

### 1. Core System (Núcleo)

O núcleo contém apenas a funcionalidade mínima necessária para o sistema operar — o "mínimo viável" que não muda entre clientes ou versões. Ele é responsável por:

- Gerenciar o ciclo de vida dos plugins (registro, carga, descarga)
- Definir e impor o **contrato** que todos os plugins devem respeitar
- Orquestrar a execução dos plugins na sequência correta
- Prover serviços compartilhados (logging, segurança, configuração)

O núcleo **não deve conter lógica de negócio específica**. Quando o núcleo começa a acumular lógica que deveria estar em plugins, o estilo degenera num monolito disfarçado.

### 2. Plugin Registry (Registro de Plugins)

O registro é o mecanismo pelo qual o núcleo descobre e carrega plugins. Pode ser implementado de várias formas:

- **Arquivo de configuração** (XML, YAML, JSON) que lista os plugins disponíveis
- **Descoberta por convenção** (o núcleo procura classes que implementam a interface em certos pacotes)
- **Registro explícito em código** (o plugin se registra no núcleo em tempo de inicialização)
- **Registry externo** (banco de dados ou serviço que lista plugins e suas versões)

A escolha do mecanismo de registro define quão dinâmico é o sistema: registros baseados em arquivo requerem reinicialização para alterar plugins; registros dinâmicos permitem adicionar/remover plugins em tempo de execução.

### 3. Plugin Contract (Contrato)

O contrato é a interface que o núcleo define e que todo plugin deve implementar. É o ponto de extensão formal do sistema. Um bom contrato:

- É **estável** — raramente muda, pois uma mudança quebra todos os plugins existentes
- É **minimalista** — define apenas o que o núcleo precisa saber sobre o plugin
- É **versionado** — permite que versões diferentes de plugins coexistam

```
Núcleo                         Plugin A          Plugin B
  │                              │                  │
  │── define ──► IPlugin ◄── implementa ────────────┤
  │                                                  │
  │── carrega A via registry                        │
  │── executa A.processar(dados)                    │
  │── carrega B via registry                        │
  │── executa B.processar(dados)                    │
```

## Formas de Implantação

Richards e Ford identificam três formas de implantar plugins no estilo MicroKernel:

| Forma | Descrição | Exemplo |
|-------|-----------|---------|
| **In-process** | Plugin carregado no mesmo processo do núcleo | Biblioteca JAR adicionada ao classpath do servidor |
| **Out-of-process** | Plugin como processo separado; comunicação via IPC ou HTTP | Serviço REST chamado pelo núcleo quando necessário |
| **SaaS** | Plugin como serviço externo na nuvem | Provedor de pagamento integrado via API |

A forma in-process é a mais simples e performática, mas exige que todos os plugins sejam implantados junto com o núcleo. A forma out-of-process permite implantação e escala independente de cada plugin, com o custo de latência de rede.

## Exemplos

### Exemplo 1 — Núcleo e Plugins em Python (In-Process)

```python
# Contrato: interface que todo plugin deve respeitar
class Plugin:
    def processar(self, dados: str) -> str:
        raise NotImplementedError

# Núcleo: gerencia o ciclo de vida e execução dos plugins
class CoreSystem:
    def __init__(self):
        self._registry: dict[str, Plugin] = {}

    def registrar_plugin(self, nome: str, plugin: Plugin) -> None:
        self._registry[nome] = plugin

    def executar(self, dados: str) -> str:
        resultado = dados
        for nome, plugin in self._registry.items():
            print(f"Executando plugin: {nome}")
            resultado = plugin.processar(resultado)
        return resultado

# Plugins: implementam o contrato sem conhecer o núcleo ou outros plugins
class PluginNormalizacao(Plugin):
    def processar(self, dados: str) -> str:
        return dados.strip().lower()

class PluginValidacao(Plugin):
    def processar(self, dados: str) -> str:
        if not dados:
            raise ValueError("Dados vazios não são permitidos.")
        return dados

class PluginAuditoria(Plugin):
    def processar(self, dados: str) -> str:
        print(f"[AUDITORIA] Processando: {dados}")
        return dados

# Uso: diferentes configurações de plugins para diferentes clientes
core = CoreSystem()
core.registrar_plugin("validacao", PluginValidacao())
core.registrar_plugin("normalizacao", PluginNormalizacao())
core.registrar_plugin("auditoria", PluginAuditoria())

resultado = core.executar("  Pedido #1234  ")
print(f"Resultado: {resultado}")
# Executando plugin: validacao
# Executando plugin: normalizacao
# [AUDITORIA] Processando: pedido #1234
# Resultado: pedido #1234
```

### Exemplo 2 — Sistema de Cálculo de Impostos (Caso de Negócio Real)

Sistemas de cálculo tributário são o exemplo canônico de MicroKernel em Richards e Ford. As regras fiscais variam por país, estado e tipo de produto — mas o fluxo de cálculo é o mesmo.

```python
from dataclasses import dataclass
from typing import Protocol

@dataclass
class Pedido:
    valor_base: float
    estado: str
    categoria_produto: str

@dataclass
class ResultadoCalculo:
    valor_base: float
    imposto: float
    valor_total: float
    descricao: str

# Contrato: todos os calculadores de imposto implementam esta interface
class CalculadorImposto(Protocol):
    def calcular(self, pedido: Pedido) -> ResultadoCalculo:
        ...

# Plugin para São Paulo
class ImpostoSPPlugin:
    def calcular(self, pedido: Pedido) -> ResultadoCalculo:
        aliquota = 0.12 if pedido.categoria_produto == "eletronico" else 0.18
        imposto = pedido.valor_base * aliquota
        return ResultadoCalculo(
            valor_base=pedido.valor_base,
            imposto=imposto,
            valor_total=pedido.valor_base + imposto,
            descricao=f"ICMS SP ({aliquota*100:.0f}%)"
        )

# Plugin para Rio de Janeiro
class ImpostoRJPlugin:
    def calcular(self, pedido: Pedido) -> ResultadoCalculo:
        aliquota = 0.20
        imposto = pedido.valor_base * aliquota
        return ResultadoCalculo(
            valor_base=pedido.valor_base,
            imposto=imposto,
            valor_total=pedido.valor_base + imposto,
            descricao=f"ICMS RJ ({aliquota*100:.0f}%)"
        )

# Núcleo: seleciona o plugin correto pelo estado
class SistemaFiscal:
    def __init__(self):
        self._calculadores: dict[str, CalculadorImposto] = {}

    def registrar(self, estado: str, calculador: CalculadorImposto) -> None:
        self._calculadores[estado] = calculador

    def calcular(self, pedido: Pedido) -> ResultadoCalculo:
        calculador = self._calculadores.get(pedido.estado)
        if not calculador:
            raise ValueError(f"Nenhum calculador registrado para o estado: {pedido.estado}")
        return calculador.calcular(pedido)

# Configuração: novo estado = novo plugin, sem alterar o núcleo
sistema = SistemaFiscal()
sistema.registrar("SP", ImpostoSPPlugin())
sistema.registrar("RJ", ImpostoRJPlugin())

pedido_sp = Pedido(valor_base=1000.0, estado="SP", categoria_produto="eletronico")
resultado = sistema.calcular(pedido_sp)
print(f"Total: R${resultado.valor_total:.2f} ({resultado.descricao})")
# Total: R$1120.00 (ICMS SP 12%)
```

Adicionar suporte a um novo estado significa apenas criar um novo plugin e registrá-lo — o `SistemaFiscal` não é modificado.

## Exemplos Modernos do Estilo

| Sistema | Núcleo | Plugins |
|---------|--------|---------|
| **Eclipse IDE** | Plataforma OSGi + runtime básico | Suporte a linguagens, ferramentas de debug, integrações SCM |
| **Jenkins** | Motor de pipeline + scheduler | Build steps, notificações, integrações com cloud providers |
| **VS Code** | Editor base + Language Server Protocol | Extensões de linguagem, temas, ferramentas de debug |
| **WordPress** | CMS base + sistema de hooks | Temas, plugins de SEO, e-commerce, analytics |
| **Sistemas de ERP** | Módulo financeiro base | Módulos de RH, manufatura, CRM por segmento de mercado |

## O Anti-padrão do Núcleo Inflado

O principal risco do estilo MicroKernel é o **Core Creep** — o núcleo cresce gradualmente com lógica que deveria estar em plugins.

```
Sintoma: o núcleo começa a ter condicionais por tipo de cliente ou região
if cliente.pais == "BR":
    aplicar_regra_brasileira()
elif cliente.pais == "US":
    aplicar_regra_americana()
```

Quando o núcleo acumula lógica específica de cliente/mercado, o estilo degenera. A solução é mover qualquer lógica variável para plugins. O núcleo deve ser **cego** ao que os plugins fazem — apenas os orquestra.

## Quando usar — e quando não usar

**Use MicroKernel quando:**
- O produto precisa ser customizável para diferentes clientes ou mercados sem alterar o código central
- Diferentes equipes desenvolvem extensões de forma independente e paralela
- Regras de negócio variam por segmento (fiscal, regulatório, regional)
- O produto é uma plataforma que terceiros estendem (ISVs, partners)

**Não use MicroKernel quando:**
- O sistema tem um único cliente e regras de negócio estáveis (o overhead não se justifica)
- As "extensões" precisam compartilhar estado entre si — isso quebra o isolamento do padrão
- A lógica é fundamentalmente hierárquica por responsabilidade técnica (considere [Camadas — 1.2](1.2%20Estilo%20em%20Camadas.md))
- O processamento é orientado a fluxo de dados sequencial (considere [Pipes and Filters — 1.3](1.3%20Pipes-filters.md))
