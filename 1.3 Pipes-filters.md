# Estilo Pipes and Filters (Pipeline Architecture)

> **Localização no mapa arquitetural:** Pipes and Filters pertence à **Família 3 — Integração e Comunicação** apresentada em *[1.1 — Mapa e Estilos de Backend](1.1%20Mapa%20e%20Estilos%20de%20Backend.md)*. Compare com [1.2 — Camadas](1.2%20Estilo%20em%20Camadas.md) quando o sistema precisar de separação por responsabilidade técnica em vez de fluxo de dados, e com [1.4 — MicroKernel](1.4%20Micro-kernel.md) quando extensibilidade via plugins for o requisito central.

## O que é o estilo Pipes and Filters?

O estilo **Pipes and Filters** — chamado de *Pipeline Architecture* por Mark Richards e Neal Ford em *Fundamentals of Software Architecture* — organiza o sistema como uma sequência de etapas de transformação de dados. Cada etapa é um componente independente chamado **filtro**; os canais que transportam os dados entre filtros são os **pipes**.

<img width="1098" alt="image" src="https://github.com/user-attachments/assets/dbf18625-fe5a-4142-b9e5-58ac0ff93ff3" />

A premissa fundamental é a **imutabilidade do contrato**: cada filtro recebe dados, os transforma e os emite — sem depender do estado de outros filtros. Filtros não se comunicam diretamente entre si, apenas através dos pipes. Isso torna o sistema altamente modular: filtros podem ser adicionados, removidos ou reordenados sem impactar os demais.

Esse estilo é a base de ferramentas que os desenvolvedores usam diariamente sem perceber: pipelines Unix (`cat | grep | sort | uniq`), pipelines de CI/CD, workflows de ETL, e sistemas de streaming como Apache Kafka e Apache NiFi.

---

## Características Arquiteturais

Richards e Ford avaliam o estilo Pipeline em *Fundamentals of Software Architecture* (Cap. 11):

| Característica | Avaliação | Observação |
|---|---|---|
| Custo geral | ⭐⭐⭐⭐⭐ | Baixo custo de entrada; conceito simples e direto |
| Simplicidade | ⭐⭐⭐⭐⭐ | Estrutura linear fácil de entender e explicar |
| Escalabilidade | ⭐⭐☆☆☆ | Filtros individuais podem escalar; o pipeline como um todo é limitado |
| Elasticidade | ⭐⭐☆☆☆ | Difícil expansão/retração rápida sob carga variável |
| Implantabilidade | ⭐⭐⭐☆☆ | Filtros podem ser implantados separadamente em versões modernas |
| Testabilidade | ⭐⭐⭐⭐⭐ | **Ponto forte:** cada filtro é independente e testável em isolamento |
| Desempenho | ⭐⭐☆☆☆ | Processamento sequencial introduz latência acumulada |
| Modularidade | ⭐⭐⭐⭐⭐ | **Ponto forte:** máxima modularidade por design |
| Confiabilidade | ⭐⭐⭐☆☆ | Falha num filtro para o pipeline; exige estratégias de reprocessamento |

> **Leitura recomendada:** Richards, M.; Ford, N. *Fundamentals of Software Architecture*, 2ª ed., Cap. 11 — Pipeline Architecture Style.

## Os Quatro Tipos de Filtro

Richards e Ford identificam quatro tipos canônicos de filtro, cada um com um papel bem definido no pipeline:

| Tipo | Papel | Exemplo |
|------|-------|---------|
| **Producer** (Fonte) | Ponto de entrada do pipeline; gera ou lê os dados iniciais | Leitura de arquivo CSV, consumo de fila, chamada a API externa |
| **Transformer** (Transformador) | Recebe dados, aplica transformação e os emite | Normalizar campos, enriquecer com dados externos, calcular métricas |
| **Tester** (Filtro/Testador) | Avalia os dados contra um critério; descarta ou encaminha | Filtrar registros inválidos, separar por categoria, validar schema |
| **Consumer** (Saída) | Ponto de término do pipeline; persiste ou envia os dados processados | Gravar no banco, publicar em tópico Kafka, enviar e-mail |

```
[Producer] → pipe → [Tester] → pipe → [Transformer] → pipe → [Consumer]
   lê CSV          filtra linhas      normaliza campos      grava no banco
                   com erro
```

Pipelines reais combinam múltiplos transformadores e testadores entre o producer e o consumer. Cada filtro pode ser executado em paralelo com outros filtros do mesmo tipo (fan-out), e um pipeline pode ter múltiplos consumers para destinos diferentes (fan-in/fan-out).

## O Anti-padrão do Filtro com Estado

O princípio fundamental dos filtros é a **ausência de estado compartilhado**. Filtros com estado introduzem acoplamento implícito e tornam o pipeline frágil e difícil de testar.

```python
# ERRADO: filtro com estado acumulado compartilhado
class FiltroCumulativo:
    total = 0  # estado compartilhado — problema

    def processar(self, valor):
        self.total += valor
        return self.total  # resultado depende da ordem de execução
```

```python
# CORRETO: filtro stateless — cada invocação é independente
class FiltroTransformador:
    def processar(self, dado):
        return {**dado, "valor_normalizado": dado["valor"] / 100}
```

Quando o pipeline requer acumulação de estado (ex: janelas de tempo em streaming), utilize mecanismos externos como Redis ou o state store do Kafka Streams — nunca estado interno do filtro.

## Exemplos

### Exemplo 1 — Pipeline Unix (Pipes nativos do sistema operacional)

O pipeline Unix é a implementação original e mais pura do padrão. Cada comando é um filtro; o operador `|` é o pipe.

**Dados de entrada (`dados.txt`):**
```
Service A,up,120
Service B,down,0
Service C,up,95
```

```bash
# Producer: cat lê o arquivo
# Tester: grep filtra apenas serviços ativos
# Tester: grep filtra por duração > 100s (simplificado)
cat dados.txt | grep ",up," | grep ",120"
```

**Saída:**
```
Service A,up,120
```

Cada comando não sabe de onde vieram os dados nem para onde vão — apenas transforma o stream que recebe.

### Exemplo 2 — Pipeline com Kafka (Producer → Filtros → Consumer)

Em sistemas distribuídos modernos, os pipes são tópicos Kafka e os filtros são consumers/producers. Cada filtro é um microsserviço ou processo independente.

**1. Producer — publica dados no tópico de entrada**
```python
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

messages = [
    {"id": 0, "valor": 0},
    {"id": 1, "valor": 10},
    {"id": 2, "valor": 20},
    {"id": 3, "valor": 30},
    {"id": 4, "valor": 40}
]

for message in messages:
    producer.send('entrada', message)

producer.close()
```

**2. Transformer — consome de `entrada`, transforma e publica em `saida`**
```python
from kafka import KafkaConsumer, KafkaProducer
import json

consumer = KafkaConsumer(
    'entrada',
    bootstrap_servers='localhost:9092',
    value_deserializer=lambda v: json.loads(v.decode('utf-8'))
)

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

for message in consumer:
    dado = message.value
    dado['processado'] = True          # transformação
    dado['valor_dobrado'] = dado['valor'] * 2
    producer.send('saida', dado)

consumer.close()
producer.close()
```

**3. Consumer — persiste os dados processados**
```python
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'saida',
    bootstrap_servers='localhost:9092',
    value_deserializer=lambda v: json.loads(v.decode('utf-8'))
)

for message in consumer:
    print(f"Persistindo: {message.value}")
    # gravar no banco de dados

consumer.close()
```

### Exemplo 3 — Pipeline de Monitoramento com Múltiplos Filtros e Kafka + MongoDB

Este exemplo demonstra um pipeline mais completo com filtros paralelos (Tester de duração e Tester de uptime) convergindo num Consumer que persiste para MongoDB.

<img width="1102" alt="image" src="https://github.com/user-attachments/assets/3eb7c776-9617-40bf-aee5-1725139627dc" />

**Producer — publica dados de serviços:**
```python
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

services = [
    {"service": "Service A", "status": "up", "duration": 120},
    {"service": "Service B", "status": "down", "duration": 0},
    {"service": "Service C", "status": "up", "duration": 95}
]

for s in services:
    producer.send('service-info', s)
producer.close()
```

**Tester de Duração — filtra serviços com duração > 100s:**
```python
from kafka import KafkaConsumer, KafkaProducer
import json

consumer = KafkaConsumer('service-info', bootstrap_servers='localhost:9092',
    value_deserializer=lambda v: json.loads(v.decode('utf-8')))
producer = KafkaProducer(bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8'))

for message in consumer:
    data = message.value
    if data['duration'] > 100:
        producer.send('duration-filtered', data)

consumer.close()
producer.close()
```

**Tester de Uptime — filtra serviços com status "up":**
```python
consumer = KafkaConsumer('service-info', bootstrap_servers='localhost:9092',
    value_deserializer=lambda v: json.loads(v.decode('utf-8')))
producer = KafkaProducer(bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8'))

for message in consumer:
    data = message.value
    if data['status'] == 'up':
        producer.send('uptime-filtered', data)

consumer.close()
producer.close()
```

**Transformer de Duração — calcula duração acumulada:**
```python
consumer = KafkaConsumer('duration-filtered', bootstrap_servers='localhost:9092',
    value_deserializer=lambda v: json.loads(v.decode('utf-8')))
producer = KafkaProducer(bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8'))

accumulated = 0
for message in consumer:
    data = message.value
    accumulated += data['duration']
    data['accumulated_duration'] = accumulated
    producer.send('duration-calculated', data)

consumer.close()
producer.close()
```

**Consumer — persiste no MongoDB:**
```python
from kafka import KafkaConsumer
import pymongo
import json

client = pymongo.MongoClient("mongodb://localhost:27017/")
collection = client['service_db']['service_data']

consumer = KafkaConsumer(
    'duration-calculated', 'uptime-filtered',
    bootstrap_servers='localhost:9092',
    value_deserializer=lambda v: json.loads(v.decode('utf-8'))
)

for message in consumer:
    collection.insert_one(message.value)

consumer.close()
```

## Quando usar — e quando não usar

**Use Pipes and Filters quando:**
- O problema é fundamentalmente um fluxo de transformações de dados (ETL, processamento de logs, validação em múltiplas etapas)
- A testabilidade isolada de cada etapa é crítica
- Diferentes equipes precisam desenvolver etapas independentemente
- O pipeline pode ser composto e recomposto dinamicamente

**Não use Pipes and Filters quando:**
- O sistema tem forte lógica de negócio hierárquica (considere [Camadas com DDD — 1.2](1.2%20Estilo%20em%20Camadas.md))
- As etapas precisam compartilhar estado mútuo — isso destrói a independência dos filtros
- A latência acumulada de cada etapa é inaceitável para o caso de uso (pipelines síncronos em tempo real têm custo alto)
- O fluxo de dados é não-linear e requer orquestração complexa (considere Arquitetura Orientada a Eventos — Capítulo 4)
